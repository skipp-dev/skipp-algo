# ADR-0006: HERO Vocabulary Discipline

**Status:** Accepted (2026-04-24)
**Author:** SMC Engineering (PR #125, audit-followup to PR #123)
**Supersedes / extends:** ŌĆö
**Related:** ADR-0003 (Pine library versioning), ADR-0005 (Pure-stdlib measurement runtime), Boundary-Contract Improvement Plan 2026-04-23 (PR-BC-04).

## Context

The HERO contract in `scripts/smc_hero_state.py` exposes 7 product-level fields consumed by both Python dashboards and Pine v5 library `smc_micro_profiles_generated.pine` (regenerated by `scripts/generate_smc_micro_profiles.py`):

| Field                | Vocab status before this ADR              |
|----------------------|-------------------------------------------|
| `HERO_TRUST`         | `HERO_TRUST_VOCAB` frozenset (5 values)   |
| `HERO_SETUP_QUALITY` | `HERO_SETUP_QUALITY_VOCAB` (4 values)     |
| `HERO_ACTION`        | `HERO_ACTION_VOCAB` (4 values)            |
| `HERO_BIAS`          | **none** ŌĆö only docstring + literal returns in `_derive_bias` |
| `HERO_MARKET_MODE`   | **none** ŌĆö only docstring + `DEFAULTS` value |
| `HERO_WHY_NOW`       | free text (catalyst phrase) ŌĆö N/A         |
| `HERO_RISK`          | partial set in `_derive_risk` (5 values, no constants) |

Two of the 5 controlled-vocabulary axes (`HERO_BIAS`, `HERO_MARKET_MODE`) lacked a formal `frozenset` constant. PR #123 (`tests/test_hero_observed_vocab_pin.py`) introduced an *observed* pin as a stop-gap and explicitly documented the gap as audit-finding **M-1**.

## Decision

Every `HERO_*` channel that is consumed by a downstream Pine library MUST expose:

1. A set of `HERO_<channel>_<value>: str` module-level string constants in `scripts/smc_hero_state.py` (one per allowed value).
2. A `HERO_<channel>_VOCAB: frozenset[str]` containing exactly those constants.
3. The corresponding `_derive_<channel>` (or equivalent) helper MUST return values from those named constants ŌĆö no bare string literals, no fallthrough `return "FOO"`.
4. A strict pin test in `tests/` that asserts:
   - the vocab `frozenset` exists and has the canonical content,
   - the helper's literal/constant returns match the vocab exactly (extra OR missing ŌåÆ fail),
   - `DEFAULTS["HERO_<channel>"]` is a vocab member,
   - any docstring enumeration is in sync.

This ADR brings `HERO_BIAS` and `HERO_MARKET_MODE` into compliance. `HERO_RISK` remains advisory text (free string) by design and is **not** in scope; if/when Pine starts gating on `HERO_RISK` literals, a follow-up ADR must extend the discipline to it.

## Consequences

### Positive

- Single source of truth: editing the vocab requires editing `smc_hero_state.py`, which trips the pin test, which forces a CHANGELOG entry + Pine-side review.
- Refactor safety: renaming a value (e.g. `"FLAT"` ŌåÆ `"NEUTRAL"`) is now a one-spot source change with a test that immediately surfaces every consumer that hard-coded the old literal.
- Pine boundary contract: a value drift on the Python side is now visible in the diff (vocab change), not buried in a deep helper return.

### Negative

- `_derive_bias` is now ~13 lines instead of 8; the indirection through named constants is mild syntactic overhead.
- Adding a 4th bias value (e.g. `"HEDGED"`) requires touching 4 places (constant + vocab + `_derive_bias` branch + test pin) instead of 1. This is intentional friction.

### Versioning rule

Any change to `HERO_BIAS_VOCAB` or `HERO_MARKET_MODE_VOCAB` (add, remove, rename) is a **breaking** change for the Pine library `smc_micro_profiles_generated` and MUST be paired with:

* `library_field_version` bump in the generated Pine library (per ADR-0003).
* CHANGELOG entry under `### Schema Versions`.
* Update of the consuming Pine dashboards (`SMC_Dashboard.pine`, `SMC_Mobile_Dashboard.pine`).

## Alternatives considered

1. **`Enum` subclass.** Rejected: Pine cannot import Python enums, and the existing 3 vocabs (TRUST / SETUP_QUALITY / ACTION) use the `frozenset[str]` pattern. Consistency wins.
2. **Pydantic `Literal` / `StrEnum`.** Rejected: introduces a typing dependency for what is fundamentally a string contract; no runtime benefit over `frozenset` membership check.
3. **Leave the observed-pin as-is.** Rejected: M-1 is the only remaining gap in the HERO vocab discipline; closing it costs ~30 LOC of source + ~50 LOC of test rewrite.
4. **Auto-generate vocabs from a YAML manifest.** Rejected (for now): premature; revisit if the number of HERO axes grows past 6.

## Migration recipe (template for future axes)

When promoting a new field to vocab discipline:

```python
# In scripts/smc_hero_state.py:
HERO_<X>_<VALUE_A>: str = "VALUE_A"
HERO_<X>_<VALUE_B>: str = "VALUE_B"

HERO_<X>_VOCAB: frozenset[str] = frozenset({
    HERO_<X>_<VALUE_A>,
    HERO_<X>_<VALUE_B>,
})

def _derive_<x>(...) -> str:
    if cond_a:
        return HERO_<X>_<VALUE_A>
    return HERO_<X>_<VALUE_B>
```

Then add a strict pin test mirroring `tests/test_hero_observed_vocab_pin.py` and bump `library_field_version` if the field is Pine-consumed.
