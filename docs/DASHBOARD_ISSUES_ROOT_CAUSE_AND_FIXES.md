## Detailed Analysis

### 1. ✅ GitHub Workflow Timeline - Overlapping Labels [FIXED]

**Symptom:** Timeline labels are overlapping and unreadable
**Panel:** "GitHub Workflow Status — Timeline" (dashboard.json:2129-2236)

**Root Cause:**
The legend format `{{workflow}}` doesn't distinguish between workflows with the same name but different trigger events. Metrics include three label dimensions (`workflow_id`, `workflow`, `event`), but only `workflow` was displayed.

**Fix Applied:**
```json
- "legendFormat": "{{workflow}}",
+ "legendFormat": "{{workflow}} ({{event}})",
```

**Impact:** Timeline now shows distinct rows for workflows triggered by different events (e.g., "deploy (schedule)" vs "deploy (push)")

### 2. ✅ GitHub Workflows Latest Run Detail - Panel Height [FIXED]

**Symptom:** Panel is scrollable, workflow details hidden
**Panel:** "GitHub Workflows — Latest Run Detail" (dashboard.json:3287-3490)

**Root Cause:**
Fixed height of 9 rows cannot accommodate all workflows when monitoring >8-10 distinct workflows.

### 3. ⚠️ Benzinga Provider - DISABLED Status

**Symptom:** Benzinga shows as "DISABLED - Provider disabled (not ingested)"
**Panel:** "News Providers — Ingest Status & Reason" (dashboard.json:3121-3267)

**Root Cause - Bootstrap Problem:**

The provider status logic in `services/live_overlay_daemon/metrics.py:640-685` marks a provider as DISABLED (state_code=3) when:
```python
is_disabled = status is not True and error.lower() == "disabled"
```

The news snapshot (`newsstack_fmp/pipeline.py:1103-1115`) sets `ok` to:
```python
"ok": _bz.fetch_total > 0 and _bz.fetch_errors == 0
```

On **first startup** before any successful fetch:
- `fetch_total == 0` → `ok = False`
- No `error` field is set → defaults to empty string
- `status is not True` but `error != "disabled"` → should show as "DEGRADED" (state_code=1), not "DISABLED"

**BUT:** If the news snapshot doesn't exist yet or `enable_benzinga_rss` is False in the signals producer environment, the provider entry is missing entirely from the snapshot, which the metrics module interprets differently.

**Action Required:**

1. **Verify signals producer configuration:**
   ```bash
   # Check if benzinga_rss is enabled in the producer
   gh api repos/skippALGO/skipp-algo/actions/secrets/ENABLE_BENZINGA_RSS
   # or check Railway environment variables
   ```

2. **Check news snapshot:**
   ```bash
   # On the Railway container or wherever live_overlay_daemon runs:
   cat artifacts/live_overlay/news_snapshot.json | jq '.providers.benzinga_rss'
   ```

3. **If missing, trigger a signals producer run** to generate a fresh news snapshot with benzinga_rss included.

4. **Expected healthy state:**
   ```json
   {
     "benzinga_rss": {
       "ok": true,
       "fetch_total": 1,
       "fetch_errors": 0,
       "items_parsed": 20
     }
   }
   ```

### 4-7. ⚠️ Daily Experiment Panels - N/A or 0 Values

**Affected Panels:**
- Daily Experiment — Snapshot Age (dashboard.json:4102)
- Daily Experiment — Files Scanned (dashboard.json:4155)
- FVG 5m Verdict (Phase E2) (dashboard.json:4204)
- BOS 4H Verdict (Phase E2) (dashboard.json:4280)
- Latest Per-Family Detail (dashboard.json:4356)
- Family Hit-Rate Over Time (dashboard.json:4469)
- Per-Day Family Hit-Rate — History (dashboard.json:4520)

**Root Cause - Missing Snapshot Files:**

All experiment panels depend on two snapshot files consumed by `services/live_overlay_daemon/compute.py`:

1. **Experiment Snapshot** (lines 507-554):
   - Path: `artifacts/live_overlay/plan_2_8_tf_family_rollup.json`
   - Or: `EXPERIMENT_SNAPSHOT_URL` environment variable
   - Generated by: Plan 2.8 benchmark/evaluation script
   - Contains: `per_tf` family rollups, `phase_e2_verdict`, `scoring_root`

2. **Experiment History** (lines 539-567):
   - Path: `artifacts/live_overlay/plan_2_8_history.jsonl`
   - Or: `EXPERIMENT_HISTORY_URL` environment variable
   - Format: JSONL (one JSON object per line)
   - Contains: Daily snapshots of `per_tf` data with `captured_at` timestamps

**Metrics Module Logic** (`services/live_overlay_daemon/metrics.py:474-567`):

```python
def _experiment_snapshot() -> dict[str, float]:
    exp = compute._load_experiment_snapshot()
    if not isinstance(exp, dict):
        return {"loaded": 0.0, "age_known": 0.0, ...}
    # Extract metrics from exp["per_tf"], exp["phase_e2_verdict"], etc.
```

When snapshot is missing:
- `loaded` = 0.0
- `age_known` = 0.0 → panels show "N/A"
- `files_scanned` = 0
- No verdict metrics emitted → panels show "N/A"

**Action Required:**

1. **Check if snapshot files exist:**
   ```bash
   # On live_overlay_daemon host (Railway container)
   ls -lh artifacts/live_overlay/plan_2_8*.json*
   ```

2. **Generate snapshots:**
   - Run the Plan 2.8 evaluation pipeline to produce `plan_2_8_tf_family_rollup.json`
   - Historical data requires periodic snapshot captures → append to `plan_2_8_history.jsonl`

3. **Configure environment variables** (if using remote URLs):
   ```bash
   EXPERIMENT_SNAPSHOT_URL=https://storage.example.com/plan_2_8_tf_family_rollup.json
   EXPERIMENT_HISTORY_URL=https://storage.example.com/plan_2_8_history.jsonl
   ```

4. **Verify daemon can load:**
   ```python
   # Test locally
   from services.live_overlay_daemon import compute
   snap = compute._load_experiment_snapshot()
   print(snap.keys())  # Should show: per_tf, phase_e2_verdict, scoring_root, etc.
   ```

**Expected Healthy State:**
- Snapshot Age: Shows hours since last experiment run (e.g., "12.3 h")
- Files Scanned: Shows count (e.g., "450")
- Verdicts: Show status codes (e.g., "measured", "insufficient_data")
- Family tables/graphs: Show hit rates and event counts per timeframe family

When report is missing:

- `age_known` = 0.0 → Age panel shows "N/A"

### 10. ✅ Railway Container Metrics - Already Working

**Status:** ✅ **No action required** - panels exist and are correctly configured

**Panels:**
- CPU Usage (rate 5m) (dashboard.json:4752-4828)
- Memory RSS (MB) (dashboard.json:4829-4894)
- Open File Descriptors (dashboard.json:4895-4944)

**Metrics:**
- `live_overlay_process_cpu_seconds_total{job="live_overlay"}` (counter)
- `live_overlay_process_resident_memory_bytes{job="live_overlay"}` (gauge)
- `live_overlay_process_open_fds{job="live_overlay"}` (gauge)

**Data Source:**
`services/live_overlay_daemon/metrics.py` lines 1398-1507 via `_collect_process_metrics()` (lines 717-794)

**Process Metrics Collection:**
- CPU: `resource.getrusage(RUSAGE_SELF).ru_utime + ru_stime`
- Memory: `/proc/self/status` (Linux) or `resource.getrusage().ru_maxrss` (macOS)
- FDs: `/proc/self/fd/` count (Linux) or `resource.getrusage().ru_maxrss` fallback

**Railway-Specific Container Metrics** (separate from process metrics):

`services/live_overlay_daemon/railway_metrics.py` polls the Railway GraphQL API for:
- `CPU_USAGE` → `railway_container_cpu_cores`
- `MEMORY_USAGE_GB` / `MEMORY_LIMIT_GB` → `railway_container_memory_*`
- `DISK_USAGE_GB`, `NETWORK_RX_GB`, `NETWORK_TX_GB`

These metrics are exposed via the daemon's `/metrics` endpoint and scraped by Grafana Alloy.

**If showing N/A:**
1. Verify Prometheus/Alloy is scraping `http://daemon:5040/metrics`
2. Check scrape job label matches `job="live_overlay"`
3. Ensure daemon has been running long enough to accumulate CPU time (>1 scrape interval)
