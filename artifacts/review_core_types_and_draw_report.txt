  Risk: medium/high. In Pine, these parameters are special value domains. Using raw `string` fields can compile in some contexts but is a recurring compile-risk in exported library type surfaces, especially when values are later passed into object constructors/setters that expect those specific domains. This is the biggest risk in the file.

- Lines 21, 35-36, 57-58: exported types store `varip chart.point` fields.

    varip chart.point end

  Risk: medium. `chart.point` inside exported types can be valid, but `varip` on complex object fields in exported types is a known compile-sensitivity area for TradingView libraries. If the editor rejects exported library types with object-handle or point fields marked `varip`, these declarations are likely suspects.

  Risk: low/medium. Overloaded methods are generally allowed, but same-name exported method overloads on different receiver types can occasionally make publish/import diagnostics harder. Not an obvious blocker by itself.

3) Concrete edit recommendations

For SMC++/smc_core_types.pine
- No edit required unless a live TradingView compile proves otherwise.
- Keep header and library name exactly as-is.

For SMC++/smc_draw.pine
Priority 1:
- Replace generic `string` field types for special style/location domains with the proper Pine parameter domain types if Pine accepts them in exported types, or otherwise normalize the library API to use plain strings only at the API edge and convert to built-ins inside draw/apply helpers.
- Most likely compile-safe fallback if TradingView rejects current declarations:
  - remove the explicit `string` type annotations from style/location/font/text option fields and let Pine infer from defaults, if allowed in type fields; or
  - define those fields using the exact domain-compatible types supported by Pine editor; or
  - if neither works, reduce exported type surface to only primitive-safe fields and move style constants resolution into non-exported helper logic.

Priority 2:
- If compile fails on exported `varip chart.point` fields, change:
  - `varip chart.point point/start/end/left_top/right_bottom`
  to
  - `chart.point point/start/end/left_top/right_bottom`
  first.
- If still failing, consider replacing stored points with primitive coordinates (`int time/index`, `float price`) and create points only at draw time. That is a bigger API change, so only do it if needed.

Priority 3:
- If constructor overload resolution errors appear, rewrite `label.new`, `line.new`, and `box.new` calls using named arguments instead of long positional argument lists. That makes style-domain mismatches much easier to isolate and is the safest remediation for TradingView compile debugging.

Priority 4:
- If direct mutation of `chart.point` fails in lines 111-124, refactor update helpers to return rebuilt points instead of mutating fields in place.
